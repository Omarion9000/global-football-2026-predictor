# 21 — Phase 9D UI

Phase 9D ships the public-facing UI for the 2026 tournament simulator. It
**replaces the Phase 6 demo front-end** (fictional Aurelia / Bellatrix
fixtures, MatchCard hero, fictional probability timeline) with three views
driven by the offline Phase 9C / 9B.2 simulator output:

1. **Title probabilities** (`/`) — hero ranking of all 48 teams.
2. **Group stage** (`/groups`) — 12-panel advancement grid.
3. **Bracket** (`/bracket`) — round-by-round leaderboards + R32→Final tree.

> **Independence and non-affiliation.** The UI is an independent statistical
> exercise. It is not endorsed by, sponsored by, or affiliated with any
> football federation, tournament organiser, or broadcaster. The disclaimer
> from `src/components/Disclosure.tsx` is rendered in the footer of every
> public page.

**Production guardrails preserved.** Phase 9D touches `src/app/**` (that is its
purpose) but does NOT change:

- the engine (`src/lib/{model,simulation,normalization,utils}`) — `MODEL_VERSION`
  stays `'v0.1.0'`;
- the database schema or persistence layer;
- the cron / scheduler route (`src/app/api/cron/predictions`) — it still uses
  `MockFixtureSource` and is out of scope for the simulator UI;
- any live-data API — there are none in this phase;
- any restricted tournament mark — see `docs/04_DATA_AND_LEGAL_POLICY.md` §3.6.

## 1. Architecture

```
┌────────────────────────────────────────────────────────────┐
│  data/raw/international_results.csv  (gitignored)          │
│         │                                                  │
│         ▼                                                  │
│  scripts/run-tournament-sim.ts  --model=confed             │
│      --write-ui-json                                       │
│      │           │                                          │
│      │           ├─► data/tournament/sim-report.json       │
│      │           │     (gitignored, full per-team agg.)    │
│      │           │                                          │
│      │           └─► src/data/tournament-sim.json          │
│      │                 (COMMITTED — UI contract)           │
│      │                                                      │
│      ▼                                                      │
│  Next.js build  ─► statically prerenders /, /groups,        │
│                    /bracket from the committed JSON         │
└────────────────────────────────────────────────────────────┘
```

The simulator runs **offline only**. The UI imports
`src/data/tournament-sim.json` at build time and statically renders the three
views. The UI never invokes the engine in-request. The cron route is left
untouched.

## 2. The data-bridge contract

The committed JSON has its TypeScript type defined in
[`src/data/tournament-sim.types.ts`](../src/data/tournament-sim.types.ts) and is
the single contract between the offline simulator and the UI.

### Shape

```ts
type TournamentSimData = {
  meta:    { generatedAt, model: 'confed', seed, n, runtimeMs,
             playedMatches, note };
  teams:   TeamOddsRow[];       // 48 rows, sorted by pTitle desc
  groups:  GroupStanding[];     // 12 groups, sorted within each by p1st desc
  bracket: BracketStructure;    // R32..Final pairs + placeholder note
};
```

`TeamOddsRow` carries `slug` (Phase 9A canonical), `displayName` (corpus
canonical), `code` (FIFA 3-letter), `iso2` (`flag-icons` lowercase 2-letter,
including `gb-eng` / `gb-sct` for the home nations), `confederation`,
`group` (A–L), and the five cumulative round probabilities `pR16`, `pQF`,
`pSF`, `pFinal`, `pTitle`.

`GroupStanding` carries `group` (A–L) and four `TeamGroupFinish` rows with
`p1st`, `p2nd`, `p3rd`, `p4th`.

`BracketStructure` mirrors `src/lib/tournament/bracket.ts`: 16 R32 matches
(each as two `BracketSlot` references), then `r16Pairs`, `qfPairs`, `sfPairs`,
`finalPair` as match-index pair arrays.

### Contract tests

[`src/data/__tests__/tournament-sim.test.ts`](../src/data/__tests__/tournament-sim.test.ts)
verifies the JSON invariants the UI assumes:

- 48 teams, sorted by `pTitle` desc;
- Σ `pTitle` ≈ 1.0 (one champion per tournament);
- cumulative round invariant `pR16 ≥ pQF ≥ pSF ≥ pFinal ≥ pTitle`;
- 12 groups A–L, four teams each;
- per-team `p1st + p2nd + p3rd + p4th = 1`;
- per-group Σ `p1st = 1` (exactly one winner);
- 16 R32 + 8 R16 + 4 QF + 2 SF + 1 Final structure;
- every R32 slot resolves to a group letter or third-place rank 1–8;
- placeholder bracket note is surfaced.

[`src/data/__tests__/flagCodes.test.ts`](../src/data/__tests__/flagCodes.test.ts)
verifies every team in the JSON has a `flag-icons` SVG asset.

## 3. The three views

### 3.1 Title probabilities — `/` (home)

The hero view. Replaces the old demo home page.

- **Masthead block**: a broadcast lower-third headline plus seed / run / N
  metadata pulled from `meta`. A stat trio shows model name (`Confed DC`),
  team count (48), and Σ P(title) as a sanity check.
- **Top-6 cards** (xl grid-6, lg grid-3, sm grid-2): flag, country name,
  confederation tag, big `P(title)` figure, a 5-segment probability strip
  (R16 → Title), and a row of round-by-round numbers. The strip widens with
  each marginal advance probability so a glance carries the shape of a team's
  campaign without reading the digits.
- **Full ranks 7 → 48 table**: every team, sorted, with its own probability
  strip alongside the precise percentages per round.
- **Methodology + limitations panel**: surfaces the model description, the
  honest limitations (host-nation under-rating, intercontinental-sample
  uncertainty, placeholder bracket), and deep links to `docs/19b` and
  `docs/20`.

### 3.2 Groups — `/groups`

12 group panels (xl grid-3, sm grid-2, mobile single-column). Each panel:

- header with the group letter at display size;
- four team rows, sorted by `p1st` desc;
- each row carries the flag, name, confederation, and an "Adv {p1+p2}" badge;
- a 4-segment advancement bar (sage 1st / butter 2nd / peach 3rd / bone 4th);
- a row of the four percentages with the 1st column accented.

### 3.3 Bracket — `/bracket`

Two sections.

**Round leaderboard**: top-5 teams most likely to reach each of R16 / QF / SF /
Final / Title. This is the honest version of the "who's in the QF" question —
it surfaces per-round reach probabilities directly, which the simulator does
track, instead of pretending to know which bracket slot a team will occupy.

**Tree**: five columns from R32 to Final.

- R32 cells annotate each slot with the team most likely to occupy it. Winner
  slots show the group's top P(1st) team; runner-up slots show the top P(2nd)
  team; third-place slots are NOT annotated with a team because the simulator
  does not currently track which of the 12 group third-placeds reach which of
  the 8 third-place bracket slots — only that 8 of the 12 advance.
- R16 / QF / SF / Final cells show the structural feed labels (`Winner R32·01
  vs Winner R32·02` etc.). This is intentional: we don't have per-slot
  probability annotations for downstream slots without tracking each
  simulation pass's bracket-position history, which is out of scope.

The page surfaces a **prominent placeholder caveat** at the top: the bracket
encodes a deterministic, plausible 32-team structure but is NOT FIFA's exact
published 2026 pairings. See [`docs/20`](20_TOURNAMENT_SIMULATOR.md) §4.4.

## 4. Live-update workflow

The simulator is designed for manual updates as real results come in. There
is **no live data feed** — that is intentional.

Workflow:

```bash
# 1. After a tournament match is played, append it to results.json:
$EDITOR data/tournament/results.json
#   add an entry like:
#   { "stage": "group", "home": "Mexico", "away": "South Africa",
#     "homeGoals": 2, "awayGoals": 0 }

# 2. Regenerate the committed UI JSON:
pnpm sim:ui
#   (equivalent to: pnpm sim:tournament --model=confed --seed=42 --n=10000
#    --write-ui-json)

# 3. Refresh the running dev server or redeploy:
pnpm dev
#   or:
pnpm build && pnpm start
```

`pnpm sim:ui` is deterministic given the same `results.json` + seed, so the
re-run is reproducible. Runtime is ~12 s on a developer laptop (fit ~9 s,
10,000 Monte Carlo passes ~3 s). Each pinned result has variance = 0; the
remaining matches re-sample.

The committed `src/data/tournament-sim.json` is the single source of truth
for the UI. The deployed UI carries whatever was in that file at build time
— a redeploy is required for an updated prediction to reach production.

## 5. Limitations surfaced in-UI

The methodology panel on `/` and the placeholder caveat on `/bracket` surface
the same five caveats documented in `docs/20` §4. Repeated here so the data
team and downstream consumers don't have to leave the UI to find them:

1. **Host-nation under-rating.** All tournament matches are modelled neutral
   in v1. USA / Mexico / Canada on home soil likely gain ~2–4 percentage
   points of title probability vs a venue-aware variant.
2. **Cross-confederation sample uncertainty.** The Phase 9B.2 confederation
   scalars are anchored by ~1,026 intercontinental matches in the corpus by
   2018, ~170 per confederation. Trust the ordering, not the decimals.
3. **Weak-data debutants.** Curaçao (CONCACAF) and Cape Verde (CAF) have
   thin top-tier corpus history; their α / δ are driven largely by the
   ridge prior.
4. **Bracket is a placeholder.** The R32 pairings are a deterministic
   structural placeholder, not the FIFA-published 2026 bracket. Replacing
   `R32_MATCHES` with the official pairings is a single-array edit.
5. **Manual `results.json` workflow.** No live API; results are entered by
   hand and the simulator re-run via `pnpm sim:ui`.

## 6. Design system — broadcast-pastel

Phase 9D adopts a "broadcast-pastel" direction: sports-broadcast structure
(strong type hierarchy, panels-as-modules, clear column rhythm, flags as
colour accents) over a pastel palette (cream/bone backgrounds, soft-but-
saturated accents, no hard black). The aesthetic intent is sophisticated
and calm, not loud.

### Typography (loaded via `next/font`)

| Role     | Family                  | Notes                                                  |
|----------|-------------------------|--------------------------------------------------------|
| Display  | **Fraunces** (variable) | Editorial weight with `opsz` + `SOFT` optical axes.    |
| Body     | **Albert Sans**         | Humanist variable sans, generous x-height for stats.   |
| Mono     | **JetBrains Mono**      | Tabular figures for the broadcast scorebox feel.       |

CSS variables `--font-display`, `--font-sans`, `--font-mono` are exposed by
`src/app/layout.tsx` and wired into `tailwind.config.ts`.

### Palette

The `bp-*` namespace in `tailwind.config.ts` and `globals.css`:

- **Surfaces**: `bp-bone` (#F4ECDC, body bg), `bp-cream` (#FBF6EA, panels),
  `bp-paper` (#FFFDF7, cards), `bp-shell` (#EFE3CB, fill base for bars),
  `bp-hairline` (#E5D8BD, borders).
- **Ink**: `bp-ink` (#2A1F1A, primary text — deep warm tone, not black),
  `bp-ink-soft` (#6B5A4B, secondary), `bp-ink-mute` (#9C8F80, kicker).
- **Pastels**: `bp-peach`, `bp-sage`, `bp-sky`, `bp-butter`, `bp-lavender`,
  `bp-rose`, plus `*-deep` variants used in the title-strip gradient.
- **Confederation tints**: `bp-conf-uefa` (sky), `bp-conf-conmebol` (butter),
  `bp-conf-caf` (sage), `bp-conf-concacaf` (peach), `bp-conf-afc` (lavender),
  `bp-conf-ofc` (rose).

### Motion

- A fixed four-corner pastel radial wash on `body.bp-page` for atmosphere.
- `bp-reveal` (staggered rise on page load) via inline `--bp-stagger`.
- Both honour `prefers-reduced-motion: reduce` per the existing global rule
  in `globals.css`.

### Probability viz

- `.bp-prob-strip` — 5-segment cumulative R16 → Title bar. Each segment
  widens with its marginal probability. Used in the top-6 cards and the
  full table.
- `.bp-group-bar` — 4-segment P(1st) / P(2nd) / P(3rd) / P(4th) bar for the
  group panels.

### Flags

`flag-icons` (MIT, public-domain SVG flags) is imported once globally in
`globals.css`. The `FlagChip` component renders a `<span class="bp-flag fi
fi-{code}">` at a configurable pixel width. `src/data/flagCodes.ts`
enumerates the slug → ISO 3166-1 alpha-2 / ISO 3166-2 mapping for all 48
tournament teams; the runner hard-errors on a missing entry, and the
contract test verifies every JSON team has a flag asset on disk.

## 7. What the UI does NOT do

- It does not invoke the engine, the simulator, or any DB client in-request.
- It does not run a server action or RSC fetch against a live data source.
- It does not surface any restricted tournament mark or federation crest.
- It does not pick up `results.json` automatically — a `pnpm sim:ui` + redeploy
  is required after every manual edit.
- It does not replace the cron / scheduler route — that path is untouched and
  remains backed by `MockFixtureSource`. Wiring the simulator into the cron is
  a separate question, deferred.

## 10. Bilingual UI — Phase 9D.1 (EN / ES)

The UI ships with a built-in English / Spanish toggle. Every piece of
interface chrome — masthead, nav, three views, methodology panel, footer,
disclaimer, table headers, bracket labels — is translatable. The simulator
JSON itself stays language-neutral: country names and confederation codes are
the same in both languages.

### 10.1 Architecture

```
┌─────────────────────────────────────────────────────────────┐
│  Browser cookie  `lang=en` | `lang=es`  (one-year max-age)  │
│         │                                                    │
│         ▼                                                    │
│  src/app/layout.tsx  (async server component)                │
│    reads cookie via next/headers                             │
│    resolves to a Lang, sets <html lang="...">                │
│    passes initialLang to LanguageProvider                    │
│         │                                                    │
│         ▼                                                    │
│  src/components/LanguageProvider.tsx  ('use client')         │
│    React context with { lang, setLang }                      │
│    setLang() also rewrites document.cookie                   │
│         │                                                    │
│         ▼                                                    │
│  Every view consumes useLang() and renders                   │
│    t(lang).path.to.string                                    │
│         │                                                    │
│         ▼                                                    │
│  Toggle (LanguageToggle) lives in the masthead. Clicking     │
│    flips the context, re-renders the tree, writes the        │
│    cookie. No page reload.                                   │
└─────────────────────────────────────────────────────────────┘
```

The pages (`/`, `/groups`, `/bracket`) became thin async server wrappers in
9D.1. Each exports a `generateMetadata` that reads the cookie and returns a
language-aware `<title>`, then renders a client view component (`HomeView`,
`GroupsView`, `BracketView`) that consumes `useLang()`. The earlier server-
rendering tree was converted to client components only where they need to
read context — server boundaries are still respected and no DB client crosses
the line.

Because the layout reads cookies, the three routes are now dynamic on Vercel
(`ƒ` in the build output). First-load JS grew from 107 kB → 114 kB to carry
the context + the two dictionaries.

### 10.2 The dictionary

[`src/i18n/dictionary.ts`](../src/i18n/dictionary.ts) is the single source
of truth. Shape:

```ts
type Dict = {
  toggle:      { ariaLabel, en, es };
  masthead:    { kicker, homeAriaLabel };
  nav:         { ariaLabel, titleProbs, groups, bracket };
  footer:      { independenceLabel, projectLabel, model(v) };
  home:        { docTitle, kickerChip, runMeta(date), …, statRunModel, …,
                 colTeam, colConf, colGroup, colR16/QF/SF/F/Title,
                 shortR16/QF/SF/F/W, … };
  groups:      { docTitle, kickerChip, meta(n), headline, sub(n), advHeader,
                 advBadge(p), pos1st/2nd/3rd/4th, aria(…) };
  bracket:     { docTitle, kickerChip, meta, headline, sub, placeholderBold,
                 placeholderBody, leaderHeadline, reachR16/QF/SF/Final,
                 winTitle, treeHeadline, colR32/R16/QF/SF/Final,
                 winnerGroup(g), runnerUpGroup(g), bestThird(n), r32Cell(n),
                 roundCell(round, n), winnerFeed(round, n), vs };
  methodology: { cardModelKicker, cardModelTitle, cardModelBodyA/B/C, …,
                 limitHostBold/Body, limitSampleBold/Body, … };
};

export const DICT: Record<Lang, Dict>;
export function t(lang: Lang): Dict;
```

Parameterised strings are typed **functions** taking their arguments inline,
so call sites are TypeScript-checked. The parity test in
[`src/i18n/__tests__/dictionary.test.ts`](../src/i18n/__tests__/dictionary.test.ts)
recursively asserts that `DICT.en` and `DICT.es` have an identical key /
function-arity shape — a key existing on one side but not the other would
fail the build, so a half-translated UI is impossible.

### 10.3 What is translated, what is not

| Translated                          | Not translated                                                |
|-------------------------------------|---------------------------------------------------------------|
| Nav, masthead kicker, toggle labels | Public product name (`Global Football 2026 Predictor`)        |
| Headlines and subtitles             | Country names (Spain, Brazil, Argentina, … same in EN + ES)   |
| Section headings + ordinals         | Confederation codes (UEFA / CONMEBOL / CAF / CONCACAF / AFC / OFC) |
| Stat-card labels, run metadata      | ISO 3166 codes, `flag-icons` lowercase codes                  |
| Table headers + column labels       | Technical terms (Monte Carlo, Dixon-Coles, MLE, ridge, Poisson, α, δ) |
| Group / bracket slot labels         | Restricted tournament marks (forbidden in both languages)     |
| Methodology + limitation bullets    | Numeric values (percentages, seed, N)                         |
| Footer chrome + model pill          |                                                               |
| Independence disclaimer             |                                                               |
| Round names (R16 → "octavos de final", QF → "cuartos de final", SF → "semifinal", Final → "final") | |

The disclaimer is the only place the word "FIFA" is allowed (per CLAUDE.md
non-negotiable rule). Both the EN and ES versions live inline inside
`src/components/Disclosure.tsx` so the vocab scan (`ui-vocabulary.test.ts`)
keeps a single canonical location.

### 10.4 Spanish judgment calls

A few choices worth noting:

- **R32 column → "Dieciseisavos de final"**, the proper Spanish football
  term (1/16 of the final). This is the standard broadcast name in Mexican
  and Latin American football. The dense `R32` abbreviation still appears
  inside tables and cell ids (e.g. `R32 · P03`) because it is universal.
  Bracket column headers use `break-words` + `leading-tight` so the long
  Spanish term wraps cleanly on two lines without disturbing the column
  rhythm.
- **Round-leaderboard kickers use the short Spanish forms**:
  "Llega a octavos / cuartos / semis / la final" + "Gana el título".
  The proper "Llega a la ronda de octavos de final" was too long for the
  card width.
- **R32 cells use `· P{nn}`** in Spanish (`P` = partido) instead of the
  English `· M{nn}`. Matches local football vocabulary.
- **Group page colour legend** translates the pastel-token names directly
  ("Verde para el 1.º, amarillo para el 2.º, melocotón para el 3.º, hueso
  para el 4.º"). This is the only place where design-system colour names
  cross into the translated copy.
- **`P(título)`** in Spanish vs **`P(title)`** in English — the only Greek
  / formula notation that gets translated. Greek letters (α, δ) and engine
  identifiers stay in source form.

### 10.5 Persistence + flash mitigation

The toggle writes a cookie:

```
Set-Cookie: lang={en|es}; Path=/; Max-Age=31536000; SameSite=Lax
```

On the next request, the server layout reads that cookie via `next/headers`
and renders the correct language for the very first HTML payload — no
English flash on returning visits. The cookie is HTTP-readable (no auth
payload, no PII) so it survives across browser sessions until manually
cleared.

If the cookie is missing or carries an unsupported value, the resolver
in `src/i18n/dictionary.ts → resolveLang()` falls back to English.

### 10.6 Boundary note

`LANG_COOKIE` and `resolveLang` live in `src/i18n/dictionary.ts` (a server-
safe module) rather than in `LanguageProvider.tsx`. A `'use client'` module
does not export plain values cleanly to a Server Component — `cookies()` was
silently reading `undefined` when the constant was imported from the client
file. The server-safe dictionary module is the canonical source.

### 10.7 Tests added in 9D.1

- [`src/i18n/__tests__/dictionary.test.ts`](../src/i18n/__tests__/dictionary.test.ts)
  — parity (deep key shape + function arity), per-string non-emptiness, banned-
  vocabulary scan in both languages.
- [`src/app/__tests__/pages.test.tsx`](../src/app/__tests__/pages.test.tsx)
  — renamed to `.tsx` so JSX is usable; now exercises each view in **both**
  EN and ES via an inline `LanguageProvider` wrapper. Asserts Spanish-specific
  strings, Spanish ordinal labels, the toggle's `aria-pressed` state in both
  languages, and that English chrome strings don't leak into the Spanish
  render (and vice versa).

The Phase 9D contract tests still apply unchanged — the data bridge is
language-neutral.

## 11. Live-state UI — Phase 9D.2

Once `data/tournament/results.json` carries one or more pinned played results
and `pnpm sim:ui` has been re-run, the UI surfaces a live state distinct from
the pure pre-tournament render. No new pages, no new tests, no engine change:
the same three views consult `meta.playedMatches` from the committed JSON and
adapt the masthead copy accordingly.

### 11.1 What flips when `playedMatches > 0`

1. **Status chip text**
   - EN: `Pre-tournament prediction` → `{N} matches played · live`
   - ES: `Predicción previa al torneo` → `{N} partidos jugados · en vivo`
2. **Status chip dot colour**: warm `bg-bp-peach` → deep `bg-bp-sage-deep`,
   a quiet visual cue that the page reflects in-tournament state.
3. **Hero subtitle paragraph**: the dictionary's `sub(n, played)` returns a
   different structure when `played > 0`, swapping the pre-tournament wording
   ("An offline Monte Carlo simulator runs the entire knockout draw…") for
   the live wording ("With **{N}** matches now pinned, the simulator
   re-samples the remaining draw…") and the emphasised noun from
   *probabilities* to *live probabilities* (EN) / *probabilidades en vivo*
   (ES). Both numbers (`played` and `n`) render in mono ink inline.
4. **`meta.note` in the committed JSON**: the runner now emits
   `"Canonical live simulator run with {N} match(es) pinned"` instead of
   the pre-tournament template when `playedForSim.length > 0`.

### 11.2 Why conditional, not replacement

The `kickerChip` and `sub` dictionary entries are functions of `played`. They
return the **pre-tournament wording when `played === 0`** and the live wording
otherwise. The implication: if `results.json` is reset to an empty `results: []`
and `pnpm sim:ui` is re-run, the JSON's `playedMatches` drops to `0` and the UI
automatically reverts to the pre-tournament copy — same code path, opposite
phase. This keeps the live-update workflow honest and reversible without
hand-editing dictionary text.

### 11.3 Updated refresh workflow

```bash
# 1. After more matches are played, append them to results.json:
$EDITOR data/tournament/results.json
#   add entries like:
#   { "stage": "group", "home": "Brazil", "away": "Scotland",
#     "homeGoals": 2, "awayGoals": 0 }

# 2. Regenerate the committed UI JSON:
pnpm sim:ui

# 3. Diff to inspect probability shifts before committing:
git diff src/data/tournament-sim.json

# 4. Boot the UI to verify the new state visually:
pnpm dev

# 5. Commit when ready (the data file is the contract; the UI re-renders).
git add data/tournament/results.json src/data/tournament-sim.json
git commit -m "Live update: <matchday|round> — <short notes>"
```

The earlier Phase 9D §4.6 (the manual `results.json` workflow) still applies.
The new convenience here is that the UI signals the live phase without any
manual text edits — the simulator's `playedMatches` count is the single
source of truth.
